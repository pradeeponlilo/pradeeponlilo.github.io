Standard Task Execution Cycle (Engineer ↔ HOD)

This flow standardizes task ownership, time tracking, review latency visibility, and continuity of work. One task has one owner at a time: Engineer in IN PROGRESS, HOD in AWAITING REVIEW. Keep work communication in ClickUp task comments (public record), and use calls/meetings only when needed—then document a brief summary in the task.

Statuses: TO DO → IN PROGRESS → AWAITING REVIEW → (IN PROGRESS or COMPLETE)
Timer Rule: ON only in IN PROGRESS; OFF in TO DO / AWAITING REVIEW / COMPLETE
Continuity Rule: After 30 minutes, pull next Primary task if available, else Secondary
HOD Target Review Time: respond within 4 working hours
If a call/meeting happens: add a short summary comment after it
Task in TO DO Owner: HOD
Work is prepared and queued. Engineer may have multiple items in TO DO, but should actively work on one at a time.
  • Task has clear description and Definition of Done
  • Priority and due date (if applicable) set
Good practice: HOD can assign 3–4 Primary tasks in TO DO for the week; engineer pulls the next one when ready.
Engineer pulls task
IN PROGRESS Owner: Engineer
Engineer is actively working. Time logged here contributes to Effective Working Hours when linked to a task.
  • Start timer when work begins
  • Stop timer when switching away
  • Prefer one active task at a time
Timer ON only in IN PROGRESS. If you must switch tasks, stop timer first, then start on the next task.
Work completed or blocked?

If the engineer cannot proceed, ownership transfers to the HOD for review/decision.

AWAITING REVIEW Owner: HOD
Engineer has completed the current step or is blocked and requires reviewer input to proceed.
  • Stop timer
  • Move status to AWAITING REVIEW
  • Add a factual comment (ready / blocked + what is needed)
Comments must be factual (event log), not opinion. Examples: “Completed, ready for review.” or “Blocked pending approval for option B.”
30-minute continuity window

Engineer does not wait indefinitely. After 30 minutes, the engineer continues work on another task.

Is another Primary Task available in TO DO?
If yes, engineer pulls the next Primary task to avoid idle time.
  • Move next Primary task to IN PROGRESS
  • Start timer on that task
This keeps output moving while the original task is awaiting review.
If NO: switch to Secondary Task
Secondary tasks keep Effective Hours meaningful during dependencies.
  • Pull Secondary task from backlog
  • Move to IN PROGRESS
  • Start timer
Secondary tasks must be pre-approved and ready-to-pick (documentation, optimization, test harness, library cleanup, etc.).
HOD Review & Decision

HOD reviews tasks in AWAITING REVIEW and converts them into an explicit decision via status change.

HOD Reviews Target: within 4 working hours
Review must result in a decision and clear next steps.
  • Check output or blocker context
  • Add clear, actionable instruction
  • Change status to COMPLETE or IN PROGRESS
If a call/meeting is needed:
It is acceptable to do a quick phone call or a short meeting to unblock faster.

Mandatory rule: After the call/meeting, add a brief summary in the ClickUp task comments so the public record remains complete.

Example comment template:
“10-min call between HOD and Engineer at 15:10. Clarified approach for X. Action agreed: implement Y, then retest. Status set to IN PROGRESS.”
Avoid private channels for decisions. Any decision that affects task scope/approach must be captured in the task comments (even if it was discussed verbally).
Status decision
Decision: COMPLETE
Task is accepted and closed.
  • No further action on this task
  • Engineer continues current work or pulls next task
If a task is COMPLETE, the engineer should not log additional time on it.
Decision: IN PROGRESS (Rework / Next step)
Task returns to engineer with instructions. Engineer resumes work when they pick it up.
  • Engineer resumes when ready
  • Start timer when work begins
  • Repeat cycle until COMPLETE
Engineer should not restart timer until they actually resume work on this task.
Notes:
  • Keep work communication primarily inside ClickUp task comments and public views for transparency and record.
  • Calls/meetings are allowed when they speed up resolution; always document a brief factual summary afterward in the task.
  • Status changes represent ownership transfer and are the primary mechanism for accountability.
